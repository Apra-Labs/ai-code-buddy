How Your API Keys Are Stored

Understanding where and how your API keys are stored is crucial for security. Here's the complete picture:

Storage Location

1. You enter API key in extension popup
↓
2. Extension validates the key format
↓
3. Stored in chrome.storage.sync
(Chrome automatically encrypts this storage)
↓
4. Only accessible to this extension
(Websites and other extensions cannot access it)

Chrome's Encryption

Chrome automatically encrypts extension storage using OS-level encryption:

  • Windows: DPAPI (Data Protection API)
  • Mac: Keychain
  • Linux: gnome-keyring or KWallet

Bottom Line: Your API keys are encrypted at rest and only accessible when you're logged into your Chrome profile.

Proof of Security: Verify It Yourself

Don't just take our word for it - verify the security claims yourself:

Test 1: Check Network Activity

  1. Open Chrome DevTools (F12)
  2. Go to Network tab
  3. Click "Send to AI" in the extension
  4. Observe: You'll see ONE request to your AI provider's API only
  5. Result: No requests to any other servers

Test 2: Review the Source Code

All code is open source and unminified. Key files to review:

  • background.js - The ONLY file that uses API keys (line ~89)
  • popup-multi.js - Where settings are saved (line ~350)
  • manifest.json - Lists all permissions and allowed domains
// From background.js - the ONLY place keys are used const response = await fetch(PROVIDER_ENDPOINT, { headers: { 'x-api-key': apiKey, // Sent ONLY to AI provider 'Content-Type': 'application/json' } }); // No other usage of API keys in entire codebase!

Test 3: Inspect Permissions

Check manifest.json for extension permissions:

{ "permissions": [ "storage", // Store settings locally "activeTab" // Access current tab only ], "host_permissions": [ "https://api.anthropic.com/*", // Claude API only "https://api.openai.com/*", // OpenAI API only "https://generativelanguage.googleapis.com/*" // Gemini only ] // NO other domains, NO "all_urls", NO broad permissions }

Open Source Guarantee: Every line of code is visible on GitHub. No compiled code, no minification, no obfuscation. What you see is exactly what runs.

Security Best Practices

For Your API Keys

  • Use separate keys: Create a dedicated API key for this extension
  • Set spending limits: Configure monthly caps in your provider dashboard
  • Monitor usage: Check your provider's usage dashboard regularly
  • Rotate regularly: Generate new keys every 30-90 days
  • Revoke if compromised: Delete and replace keys immediately if exposed

For Maximum Security

  • Use read-only keys: If your provider supports it (limits damage if leaked)
  • Enable Chrome profile password: Adds extra layer of protection
  • Keep Chrome updated: Security patches are critical
  • Use Ollama for sensitive code: 100% local processing, zero network risk
  • Load from source: Clone the repo and load unpacked for maximum control

For Your Privacy

  • Choose Ollama: For completely private, local AI processing
  • Review provider policies: Understand how each provider handles data
  • Avoid sensitive data: Don't send credentials or private keys to cloud AI
  • Use Azure for enterprise: Better compliance and data governance

Potential Security Risks & Mitigations

Being transparent about potential risks and how to mitigate them:

Scenario 1: Extension Compromise

Risk: If the Chrome Web Store version gets compromised

Mitigation:

  • Load from source (GitHub) instead of Chrome Web Store
  • Review code updates before installing
  • Chrome reviews all extensions before publishing
  • Use read-only API keys if available

Scenario 2: AI Provider Breach

Risk: Your chosen AI provider (Claude, OpenAI, etc.) gets hacked

Mitigation:

  • Set rate limits in provider dashboard
  • Use separate keys for different tools
  • Monitor usage regularly
  • Rotate keys monthly
  • Use Ollama to avoid cloud providers entirely

Scenario 3: Local Computer Compromise

Risk: Malware on your computer

Mitigation:

  • Use antivirus software
  • Keep OS and software updated
  • Use Chrome profile password
  • Note: No extension can protect against system-level compromise
